# ADR-001 Quality First Development

Date: 2025-06-18
Status: Accepted

---

## Context

Imaging Agriculture はスモールチームで開発する。

スモールチームが最も速度を失う原因は
品質の後退による回帰バグの修正コストである。

「速く作って後で直す」戦略は
初期は速く見えるが
時間とともに開発速度を指数的に落とす。

AgriPath（先行プロダクト）での経験から
以下の問題が発生した。

* 回帰バグが本番に混入した
* 仕様との逸脱が発見されるまで時間がかかった
* 品質低下が機能追加速度を落とした

---

## Decision

「品質が速度を生む」を開発方針として採用する。

具体的には以下を定める。

### 1. テストファースト

Playwright による E2E テストを
機能実装と同時に作成する。

「後でテストを書く」は禁止とする。

### 2. CI ゲート

GitHub Actions の全チェック（lint / typecheck / build / playwright）が
green にならない限りマージしない。

例外なし。

### 3. 仕様先行

コードより仕様ドキュメントを先に更新する。

QUALITY_GATE.md のチェックリストを
PR 作成前に確認する。

### 4. Sentry 即日対応

本番エラーは発生当日中に原因を特定する。

「後で調べる」は禁止とする。

### 5. 非目標の明示

品質ゲートは MVP の速度を下げるためではない。

MVP フェーズで不要な品質施策は明示的にスコープ外とする。

不要なもの（MVP 時点）

* ユニットテストのカバレッジ計測
* パフォーマンステスト（Lighthouse CI 等）
* アクセシビリティ自動テスト
* ビジュアルリグレッションテスト

---

## Consequences

### ポジティブ

* 回帰バグを自動検出できる
* 仕様とコードの乖離を PR 単位で防げる
* 本番エラーの検知が即時になる
* テストが仕様のドキュメントとして機能する

### トレードオフ

* 初期の機能実装にテスト作成コストが加わる
* CI の実行時間（目安：3〜5 分）が PR マージまでに必要になる

### 判断

このトレードオフは受け入れる。

初期コストは回帰バグの修正コストより必ず小さい。

---

## Alternatives Considered

### 「まず動かして後で品質を上げる」

採用しなかった理由

AgriPath での経験が示すとおり
「後で」は来ない。

技術的負債は複利で増える。

### 「ユニットテストを中心にする」

採用しなかった理由

MVP フェーズでは仕様変更が頻繁に起きる。

ユニットテストは実装詳細に依存するため
仕様変更のたびにテストを書き直すコストが高い。

E2E テストはユーザー導線を保護するため
仕様変更に対して強い。
